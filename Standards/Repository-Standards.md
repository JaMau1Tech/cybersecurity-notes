# Repository Standards

**Version:** 1.0

**Status:** 🔒 LOCKED

**Last Updated:** 2026-07-11

This document defines the universal repository standards used throughout the Root & Repository project.

Repository-specific structures are defined within their own workflow standards.

No changes may be made without explicit user approval.

---

# Core Principles

- Every repository follows its own documented workflow.
- Every repository maintains a consistent internal structure.
- Documentation is organized, readable, and reproducible.
- Git is used for version control on every repository.
- Repository structures are never changed without explicit approval.
- Workflow standards take precedence over assumptions.

---

# Repository Naming

Repositories use descriptive names.

Examples:

- TryHackMe
- Career
- Investigations
- Cybersecurity-Academy
- Linux
- Networking
- Projects

Avoid vague or abbreviated names.

---

# Repository Documentation

Every repository should include documentation appropriate for its purpose.

Examples include:

- README.md
- task-notes.md
- investigation-notes.md
- exam.md
- results.md

The exact documentation requirements are defined by the repository's workflow standard.

---

# Repository Structure

Repository structures are NOT defined here.

Each repository maintains its own locked structure documented within its corresponding workflow standard.

Examples:

- TryHackMe-Workflow.md
- Investigation-Workflow.md
- Cybersecurity-Academy-Workflow.md
- Career-Workflow.md

This prevents one repository's structure from affecting another.

---

# Documentation Standards

Documentation should:

- Use Markdown
- Follow the appropriate workflow standard
- Be reviewed before committing
- Use descriptive headings
- Remain consistent throughout the repository

---

# Screenshot Storage

Screenshots are stored according to the repository's workflow standard.

Screenshot folder locations are not standardized across all repositories because each repository serves a different purpose.

---

# Git Workflow

Every completed activity ends with:

1. Repository review
2. Git status
3. Git add
4. Git commit
5. Git push

The assistant provides the exact Git commands.

---

# Repository Review Checklist

Before considering work complete:

- [ ] Repository follows its workflow standard
- [ ] Documentation complete
- [ ] Naming conventions followed
- [ ] Screenshots stored correctly
- [ ] README(s) updated (if applicable)
- [ ] Markdown reviewed
- [ ] Git commands prepared

---

# Assistant Responsibilities

The assistant must:

- Follow the repository's locked workflow.
- Never assume repository structures.
- Never modify repository organization without approval.
- Reference the appropriate workflow standard.
- Review documentation before completion.
- Provide Git commands after documentation is complete.

If a repository workflow conflicts with this document, the repository workflow takes precedence.
